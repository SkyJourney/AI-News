## 关键架构决策（执行阶段落地的）

### D1 · QuartzPageTypePlugin 路径

**决策**：写 6 个自定义 pageType 用 `config.plugins.pageTypes.unshift(...)` 注册（priority: 100 > core ContentPage: 0）。

**背景**：Explore 阶段发现 `dispatcher.ts:22-37` 硬编码 `pageType.body(undefined)`，`byPageType.X.pageBody` 完全无效——唯一路径是走自定义 QuartzPageTypePlugin。

### D2 · lumina-components/ 子目录结构

**决策**：从 P3 的 3 组件扁平结构升级到 5 子目录：
```
shell/    (外壳)
shared/   (UI 原语 · 12 组件)
bodies/   (pageBody · 6 组件)
pageTypes/ (pageType 声明 · 6 文件)
enhance/  (双链增强 · 1 文件)
util/     (helpers · 4 文件)
```

**背景**：4500 行代码扁平放不合理；子目录让新人 review 时能快速定位职责。

### D3 · SHARED_CSS 聚合模式

**决策**：`shared/index.ts` export `SHARED_CSS = [shared 组件的 .css 属性].join("\\n\\n")`；6 body 组件在 `.css` 里 `SHARED_CSS + \`.lumina-<body> { ... }\``。

**背景**：Quartz 5 的 `getComponentResources` 只扫顶层 pageBody 的 css 静态属性，**不递归收 body 里嵌套的 shared helper**。若不 aggregate，shared 组件的 CSS 完全不会进最终 CSS（HTML 有 class 但没样式）。

**权衡**：6 个 body 各挂一份 SHARED_CSS aggregate 会让 Quartz 生成 6 个大的 component-css 文件（重复 shared 内容）。但每页只加载对应 body 的 aggregate（约 40 KB），依然可接受。未来若发现 CSS 体积问题，可把 shared CSS 移到 custom.scss 全局单份。

### D4 · vault 目录 slug 全小写

**决策**：所有 pageType.match / util.dirOf / Dock.href 都用 lowercased slug（`10-daily` 而非 `10-Daily`）。

**背景**：Quartz slugify 会 lowercase 所有 slug（`10-Daily/2026-07-01.md` → slug `10-daily/2026-07-01`）。P4 首次 build 时因大写不匹配所有 pageType.match 都 miss，页面 DOM 无 Lumina。改小写后立即命中。

### D5 · AI 判定按目录（+ frontmatter.type 加强）

**决策**：`60-originals` / `10-daily` / `50-zettel` → AI；`20-topics` / `40-deep-dives` → human；`frontmatter.type === "source-original"` 强化 AI 判定。

**背景**：vault 里 md 没有显式 `ai_generated` 字段，按目录分派是可靠的（自动 writer / originalizer 产出 vs 人工汇编）。未来加显式字段可覆盖目录判定。

### D6 · 40-Deep-Dives 空目录 · Virtual Page 生成

**决策**：`LuminaDeepDivesPageType.generate()` 主动返回 `{slug: "40-deep-dives/index", ...}` virtual page；LuminaDeepDivesBody 检测 `isTreeEmpty(tree)` 走 EmptyState 分支。

**背景**：40-Deep-Dives 是 v2 阶段预留（只有 .gitkeep）。folder-page 不为空目录生成 virtual index。走 generate 后 v2 后目录有 md 时依然 fallback 到 htmlToJsx 渲染（body 一样代码，双态兼容）。

### D7 · Related Intelligence = LuminaBacklinks

**决策**：`shared/RelatedIntelligence` 承担全部反向链接分栏渲染逻辑；`enhance/LuminaBacklinks` 只是薄包装（提供合理 defaults）。

**背景**：设计原本要写 GroupedBacklinks + PopoverPreview 两个组件；实际 RelatedIntelligence（按目录分组 + AI 标记 + 空态提示）已覆盖 GroupedBacklinks 全部需求，且能复用到 Sources 右栏。PopoverPreview 留 F2.4+ 或 F2.5（Quartz enablePopovers 内置基础 hover 预览，只是视觉不 Lumina）。

## 遗留问题

### R1 · Dev Server 端口冲突

Docker 占了 8080-8083 + 8801。Quartz `--serve` 默认 8080 挂。用户手动跑需要 `--port <闲置端口>` 指定。或者停 Docker 服务（不推荐 —— 可能影响其他项目）。

### R2 · Component CSS 复制 · SHARED_CSS 每 body 一份

每 body 组件挂一份 SHARED_CSS aggregate，Quartz 会生成 6 个大 component-*.css（重复 shared 内容）。总体积可接受但不最优。**F2.5 之前可优化**：把 shared CSS 迁移到 `quartz/styles/custom.scss` 全局单份。

### R3 · Popover Preview 未实现

Lumina 设计里"hover 双链弹卡片摘要"这个交互，当前用 Quartz 内置 enablePopovers（配置 yaml 里 true）。样式跟 Quartz 默认，非 Lumina 玻璃拟态。若需 Lumina 化预览，写 LuminaPopover 组件替换（预计 1-2h · 后续 F2.5）。

### R4 · 首页 Bases 视图入口卡链接

Dashboard 的 4 张 Bases 视图入口卡链接 (`_base-latest-daily` 等) 指向 Quartz 生成的 Bases 页面。**未验证**这些 slug 实际能否命中（Quartz 5 bases-page 插件生成 `.base.html`，Slug 可能是 `_base-latest-daily` 或 `_base-latest-daily.base`）。P4.6 未测。用户第一次访问可能 404。快速修：改 href 加 `.base` 或看实际 build 出来是啥。

### R5 · Sources 页 Article Progress 交互

Sources body 的右栏进度条用 `afterDOMLoaded` 挂 scroll listener。**未在浏览器实测**是否流畅（依赖 dev server）。逻辑简单，出错概率低。
