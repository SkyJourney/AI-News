# 核心转储流行病学：修复一个 18 年前的 bug

> 原文：[Core Dump Epidemiology: fixing an 18-year-old bug](https://openai.com/index/core-dump-epidemiology-data-infrastructure-bug) · openai-rss · 2026-07-01
> 抓取：2026-07-01T23:29:35+08:00 · 翻译：haiku · 8140 字

OpenAI 的模型和代理越来越依赖可扩展的数据基础设施，以便在推理时搜索相关数据：当模型在思考你的问题时。我们的一些服务用 C++ 编写，这门语言对系统的低级控制让我们能最大化性能并最小化内存使用。随着规模扩展，这些效率优势尤为重要，但 C++ 缺乏内存安全性意味着 bug 可能导致程序崩溃，因为写入了不正确或不存在的内存地址。

几个月前，我们在 Rockset 服务内部观测到一些崩溃，这是我们 ChatGPT 数据基础设施的一个定制部分，对许多数据插件和对话搜索至关重要。在每次崩溃中，一个看似正常的 C++ 函数似乎完成了执行，然后返回到一个伪造的地址，导致内核停止程序，因为指令指针不再指向代码。有时栈帧中的返回地址槽位是 NULL。有时栈指针 CPU 寄存器本身似乎偏移了 8 字节，好像 `%rsp` 在正常执行中间被某种方式递减了。在两种情况下，崩溃都发生在返回时。

这些不是应用代码的正常故障模式。一个只落在保存返回地址上的杂散写入是可能的，但极其罕见。一个在没有内联汇编、`setcontext` 或 `longjmp` 参与的情况下使 `%rsp` 错位 8 字节的 bug 就更奇怪了（我们都不用），因为编译的代码只在函数序言和尾声中直接调整这个寄存器。我们（或 ChatGPT）想到的每个假设都有强烈的反对证据，所以这个 bug 似乎不可能存在。

我们最初认为是一个问题最终变成了两个无关的 bug，巧合地在同一时间被发现。第一，Azure 主机上的静默硬件损坏，CPU 只是没有正确执行数学运算。第二，GNU libunwind 中一个 18 年前的竞态条件，这是一个在广泛使用的开源库中未被注意的 bug。

这篇文章讲述了我们如何通过像流行病学家那样思考，并构建一个关于整个崩溃人口的高质量数据集，来识别和修复看似无法解释的崩溃的故事。

## 第一次调试尝试：仔细检查几个核心转储

首先，让我们深入了解 Rockset。它是一个用于搜索和实时分析的云原生数据系统，我们在 OpenAI 的许多内部用例中使用它，例如同步连接器（Rockset 在 2024 年被 OpenAI 收购）。流式更新用于维护工作区知识库的最新索引，使 ChatGPT 能够在回答问题或执行操作时搜索相关信息。

Rockset 的执行层用 C++ 编写。C++ 语言提供对 CPU 的低级访问，这对性能和效率有利，但这意味着应用 bug 可能导致无效的内存访问和段错误。为了帮助追踪这些，我们使用 folly 的致命信号处理器在崩溃时记录栈跟踪，并将相应的核心转储（程序崩溃时状态的快照）上传到 Azure blob 存储以供稍后分析。Rockset 的所有查询处理叶节点都是复制的，这最大程度地减少了崩溃对客户端的影响。然而，每个段错误都对应一个需要修复的 bug，以满足我们的可靠性和质量目标。

我们最初的方法是像对待传统调试问题一样对待这些核心：非常仔细地检查几个核心转储，形成假设，逐一排除。

大多数崩溃发生在一个名为 `DocumentTree::updateDocument` 的方法中。在这些崩溃中，看起来 `updateDocument` 调用了某个未知函数 X，当 X 活跃时栈被破坏，然后 X 返回到一个不是可执行代码的地址。在某些情况下，X 刚弹出的栈帧看起来有效，除了其保存的返回地址是 NULL。在其他情况下，栈指针本身看起来错误，但下一个有效的栈帧仍然似乎是 `updateDocument`。

我们不知道栈何时被破坏，这留下了一个巨大的搜索空间。`updateDocument` 是一个大型方法，经过大量内联，所以 X 的候选者数量令人难以招架。

这是我们 C++ 代码中的 bug 吗？编译器或链接问题？我们某个运行时库中的问题？Linux 内核中关于信号传递或上下文切换的 bug？更稀有的东西？如果这是一个杂散写入，为什么它没有被我们的 ASAN 暂存环境捕获？

我们试图使用应用级日志来识别问题的所有出现，但栈破坏 bug 很难从日志单独分类，因为记录的栈跟踪本身是破坏的或缺失的。我们无法构造一个既没有假正例也没有假负例的日志查询。我们手动检查了更多核心转储，找到了一些额外的例子，但这个过程太费力了，无法给我们一个值得信赖的数据集。

在调查的这个阶段，我们（不正确地）排除了硬件 bug，因为我们在多个地区和多种硬件类型中看到崩溃，所以我们仍在寻找纯软件原因。有几天，我们对一次单个错位的 `%rsp` 崩溃进行了超级深入的调查，使用栈和寄存器内容重建了崩溃前的历史。这产生了一些可能的线索，但因为我们没有放弃最初的结论即所有 bug 有相同的原因，这没有让我们脱困。

## 栈的线索

在到达我们调查的转折点之前，重要的是解释我们从核心文件中提取了什么样的信息。

Rockset 用 `-fno-omit-frame-pointer` 编译，所以活跃栈帧总是通过 `%rbp` 可达的，调用者形成一个栈帧指针的链表。

在 Linux `x86_64` 上，AMD64 System V ABI 也在 `%rsp` 下方保留 128 字节作为红区（red zone）。该区域对用户空间代码可用，重要的是，作为 ABI 契约的一部分，内核承诺在传递信号时不会对其进行破坏。

红区对我们调试返回后崩溃至关重要，因为它保留了返回前的一些信息。当触发 `SIGSEGV` 时，folly 的致命信号处理器在崩溃线程的栈上运行。不再活跃的栈帧（因为它们的函数已返回）将被信号处理器破坏，除了最后 128 字节。这就是为什么我们可以说"X 刚弹出的栈帧看起来有效，除了一个 NULL 返回地址"。红区保留了一些非活跃栈帧，或有时只是一个非活跃栈帧的尾部。

我们发现了一个错位栈崩溃，其中所有涉及的函数都很小。这让我们看到 `%rsp` 在一个相对简单函数的执行中变得错位，之后有更多调用成功了。程序只在活跃函数最终试图返回时崩溃。这些代码路径都没有使用异常、内联汇编、`setcontext` 或 `longjmp`，所以如果栈指针真的按核心所示改变了，用户空间代码中就没有合理的 bug 解释这个问题。

这把我们推向了内核。

Rockset 使用信号的激进程度超过大多数程序。查询执行被分解为许多交换数据的轻量级任务。这对于高 QPS 工作负载的高效处理很重要，但它使每查询 CPU 会计很尴尬，因为许多查询的工作被多路复用到相同的线程池。

我们的解决方案是我们称之为 `coarse_thread_cputime_clock` 的东西，它廉价地近似 `clock_gettime(CLOCK_THREAD_CPUTIME_ID, ...)` 以至于可以在每个任务边界采样。`timer_create` API 可用于根据几个时间流逝的概念安排周期性信号传递，包括 CPU 时间的累积。我们安排一个信号（SIGUSR2）每隔几毫秒 CPU 时间传递一次，此时信号处理器更新一个线程局部值。即使许多任务在执行时看不到粗时钟前进，对所有增量求和会产生查询实际 CPU 时间的无偏估计。

因为我们传递信号如此频繁，一个关于上下文切换或信号传递的稀有内核 bug 似乎是合理的。我们花时间阅读 bug 报告、内核源代码和 Azure 特定的内核补丁。我们尝试了压力测试。我们找不到任何看起来相关的东西。

此时，我们决定退一步尝试不同的方法。

## 医生还是流行病学家？

调试这样的问题有两种广泛的方式。

一种是像一种医生那样行动：专注于一个患者，进行大量测试，试图从详细证据诊断单个案例。

另一种是更像流行病学家那样行动：看整个人口，问是否有单个案例无法揭示的模式。bug 是在特定发布时开始的吗？它与一个硬件 SKU（特定 CPU 和服务器型号）、一个地区或一个内核版本相关吗？是否有多个不同的集群隐藏在看起来像一个综合征的东西内？

我们大多数情况下一直处于医生模式。关键转变是决定我们需要收集高质量的人口数据。

## 清理数据

我们之前自动查找所有问题实例的尝试失败了，因为我们试图在日志上使用文本搜索。核心转储本身有很多更多信息，但手动查看它们不可扩展。我们决定投入力量构建一个可以自动分析核心转储的管道。

我们让 ChatGPT 写了一个脚本，下载每个核心文件的前缀，提取寄存器，使用日志过滤已知的假正例，并自动将崩溃标记为返回到 null、错位栈或其他。然后我们在以前生产 Rockset 核心转储的整个过程中并行运行该脚本。

这是转折点。

一旦我们有了一个干净的数据集，相关性立即出现。我们一直在将其视为一个奇怪 bug 的东西实际上是_两个_不同的崩溃人口。

返回到 null 核心分散在许多集群和地理区域。它们的频率最近增加了，但没有清晰的开始日期和干净的基础设施边界。

错位栈崩溃看起来完全不同。它们全部来自一个地区，有清晰的开始日期，从未发生在运行很久的节点上。即使它们涉及多个 Azure VM（云中托管的虚拟机），这个模式看起来像一个物理机器有坏硬件导致任何碰巧落在上面的 VM 的问题。

那是我们意识到我们一直在精神上混淆两个 bug 的时刻。因为我们混淆了两个 bug 的反例，我们找不到一个单一的连贯解释。

## Bug #1：坏主机

武装了清晰的 Kubernetes 节点列表和时间戳，我们能够追踪错位栈崩溃回到一个物理主机，这很容易被否认列表。

我们无法在控制环境中重现该主机上的寄存器损坏，即使在几周的压力测试之后。然而，一旦问题主机被撤出服务，错位栈崩溃就消失了。

删除坏主机在"阻止相同问题的新发生"的意义上不是永久解决方案。但是，我们可以改变软件，使得如果类似问题再次发生，它可以被轻松检测和处理。我们改进了我们的致命信号处理器以包括寄存器状态，使得我们只能从日志检测再发生（不需要核心转储）。我们改变了控制平面，使得 VM 通常被重用而不是回收，这使得在我们基础设施栈的级别上坏节点检测容易得多。我们也更新了我们的运行手册（和我们团队的心理模型）以包括这种可能性。

分离坏主机崩溃后，剩余的返回到 null 核心变得更容易推理。早些时候，我们排除了异常展开，因为我们认为我们有反例：在绝对不使用异常的代码路径中崩溃。但那些反例全部来自硬件损坏集群。

一旦我们用这个想法重新考虑剩余核心，我们发现这个结论完全反向：崩溃都发生在异常展开期间。

## 异常处理是动态控制转移

当 C++ 抛出一个异常时，运行时必须发现哪个 catch 块应该接收它以及哪些析构函数或清理处理器应该沿途运行。编译器发出这个元数据，但实际匹配发生在运行时。

异常展开实际上不是由调用 `throw` 的函数执行的，而是由产生的编译代码调用的辅助函数执行。那些运行时例程检查栈，获取关于在栈中找到的函数的元数据，动态查找清理处理器和 catch 块，然后将控制转移到那些位置之一。转移控制包括展开所有干预的栈帧（包括辅助函数的那些）。

操作上，这更接近一个 `longjmp` 或纤维切换而不是一个正常调用和返回。被调用方保存寄存器必须被恢复，以及栈帧寄存器 `%rbp` 和 `%rsp`。

我们的二进制链接对两个包含执行 C++ 异常展开的函数实现的库：libgcc 和 GNU libunwind。GNU libunwind 的定义被动态链接器选中。这令我们惊讶；我们预计 libgcc 实现会赢因为符号版本规则；然而，检查运行二进制显示这不是情况。

## 撤销最后一个假设

此时我们的工作假设改变，当我们放宽另一个假设我们在认为只有一个 bug 时做的。

也许我们不是看到一个函数的正常返回到 NULL。也许我们看到一个展开转移——实际上是一个 `setcontext` 风格的寄存器恢复——其中目标指令指针在控制转移前变为 NULL。换句话说，来自展开库的不正确数据而不是栈上不正确的返回地址槽。

那大大缩小了问题。要么 GNU libunwind 在计算错误的目标状态，要么它在计算正确状态和某物在它可以被应用前破坏它。

我们读了 GNU libunwind 源码，找到了它合成一个 `ucontext_t` 在栈上，用清理处理器的栈帧填充所需的寄存器状态，然后把指针传给该结构体给一个内部汇编例程：`_Ux86_64_setcontext`。

此时我们有了所有部分。

合成的 `ucontext_t` 活在一个被 `_Ux86_64_setcontext` 展开的栈帧中，在那个函数的执行期间。`_Ux86_64_setcontext` 是否在改变 `%rsp` 后从结构体读取，此时结构体不再是活跃栈的一部分？那会使它容易被信号传递破坏，例如我们频繁的 `SIGUSR2`。

## Bug #2：libunwind bug

答案是肯定的。

这是 GNU libunwind 版本中 `_Ux86_64_setcontext` 的最后六条指令，我们使用的主要由从内存到目标寄存器的 `mov` 指令组成：

```
movq $UC_MCONTEXT_GREGS_RSP(%rdi), %rsp
movq $UC_MCONTEXT_GREGS_R8(%rdi), %r8
movq $UC_MCONTEXT_GREGS_R9(%rdi), %r9
movq $UC_MCONTEXT_GREGS_RBX(%rdi), %rbx
movq $UC_MCONTEXT_GREGS_RIP(%rdi), %r8
retq
```

（`%rdi` 指向栈分配的 `ucontext_t`，而 `UC_MCONTEXT_*` 宏只展开到一个特定寄存器被存储的固定偏移。）

第一条指令是竞态窗口的开始。它更新 `%rsp` 以指向活跃栈的新底部。就在这发生时，由 `%rdi` 指向的结构体不再是活跃栈的一部分（或红区），它不再对内核限制。

通常这不会导致问题，但如果一个信号在完全正确（错误？）的时刻到达，内核将在 `%rsp-128` 处建立信号帧。那可以覆盖由 `%rdi` 指向的内存。

如果那发生在下一条指令读 `UC_MCONTEXT_GREGS_RIP(%rdi)` 前，那么恢复的指令指针可能被损坏。在我们的崩溃中，它变为 NULL。

那是 bug。

## 为什么核心掩饰为常见坏返回

这个汇编也解释了让我们困惑的观察之一：为什么函数 X 在前面栈帧的返回地址槽中有 NULL。

`setcontext` 被编写来恢复所有寄存器，包括 `%rdi`，所以它不能在最终控制转移的时刻使用那个寄存器读 `UC_MCONTEXT_GREGS_RIP(%rdi)`。相反，它早些读取值，保存到栈，恢复少数更多寄存器，然后使用 `retq` 读取保存的值并转移控制。

在核心中看起来像"一个函数返回到 NULL"的实际上是"展开器合成了栈上的目标返回地址，但那个目标在转移完成前被损坏"。我们假设返回地址槽中的损坏必须发生在位置，因为我们不知道有什么地方（可损坏的）数据被有意写入返回地址槽。

## 单指令竞态窗口

使这个 bug 看起来荒谬的是这个竞态窗口有多窄。在这种竞态条件中，外部事件（信号）需要发生在另一个线程采取的两个步骤之间。这些步骤彼此越接近，竞态条件发生的可能性越小。

在这个情况下，脆弱窗口字面上是一条指令宽！一个信号必须在 `%rsp` 改变后但在下一条指令加载 `%rip` 前被传递。在现代超标量乱序 CPU 上可以每个周期运行几个简单指令，所以竞态窗口大约是一百皮秒。

当我们找到这个竞态时，我们的第一反应是它必须太罕见来解释观察到的崩溃率。我们跨舰队看到超过十来个返回到 null 的崩溃每天。一个单指令竞态在异常清理期间真的可以解释那个吗？

我们求助于费米估计。如果脆弱窗口在 $10^{-10}$ 秒的数量级，而 SIGUSR2 每 $10^{-2}$ 秒 CPU 时间到达，那么每个异常清理处理器或 catch 块大约有 $10^{-8}$ 的概率失去竞态。

Rockset 使用异常作为其内部摄取背压机制的一部分。单个超载主机可以每秒抛出大约 $10^{4}$ 个异常。这意味着使用背压的主机平均故障间隔时间是 $10^{4}$ 秒，或每几小时一次崩溃。在舰队规模，那已经足够解释观察到的崩溃频率。

## 为什么 libunwind bug 现在出现？

GNU libunwind bug 是旧的——超过 18 年前，出现在第一个支持 C++ 异常展开的 `x86_64` 版本中。

那它为什么现在出现？

崩溃率大约与抛出多少异常和传递多少信号成正比。它也依赖信号处理器消耗多少栈。

Rockset 在所有三个轴上都不寻常。我们以高率抛出异常作为正常超载控制的一部分；我们因为 `coarse_thread_cputime_clock` 而异常频繁地传递 `SIGUSR2`；而且今年早些我们通过添加 `timer_getoverrun` 的调用使 `SIGUSR2` 处理器使用更多栈，所以我们可以解释合并信号。

那最后的改变似乎是重要的。如果处理器使用很少的栈，它可能不会到达和覆盖陈旧的 `ucontext_t` 内存。在那个改变前，我们根本没有观察到这些崩溃。改变后，在我们对使用背压机制压力的一些用例加载直到费率保持低。

换句话说，libunwind bug 一直在那里，但我们的异常率、信号率和处理器栈使用的乘积只最近越过了它变成操作上可见的阈值。

这个机制也解释了巧合，即硬件 bug 和 libunwind bug 大多数在 `DocumentTree::updateDocument` 内崩溃。来自 libunwind 的崩溃严重偏向这个方法，因为当我们抛出异常应用摄取背压时它总是活跃的。它也对 `%rsp` 错位崩溃被重度选择，因为坏硬件节点是我们用于散装摄取的 SKU，它花费大多数 CPU 时间在那个方法。

我们的直接缓解是从 GNU libunwind 切换到 libgcc 的展开器。那本身是一个好的权衡：libgcc 的实现受益于大量工作以减少锁争论，这在扩展到大 VM 时很重要。

我们也上溯了一个自包含的重现器和一个[修复](https://github.com/libunwind/libunwind/commit/a9b9293b286c14b9ed19db501aa347b46edd8a28)到 GNU libunwind，并验证了其他展开器没有类似问题。

## 人口级诊断的力量

这个调试旅程教会了我们很多关于动态链接、DWARF 展开元数据、Linux 信号传递、System V ABI 和 C++ 异常机制的具体细节。但主要课程比所有这些更简单。

最重要的步骤不是聪慧的汇编阅读或深知识的细节。它是构建一个高质量的数据集。在没有这个数据集的情况下，我们将两个不同的现象混淆到一个故事中，并试图用推理方式摆脱混淆。一旦我们有了准确和完整的人口数据，问题的结构变得明显：一个崩溃人口属于坏主机，而另一个属于 libunwind 的竞态。一旦数据变更，调试变更了。

对于像 Rockset 这样的基础设施系统，那很重要。这个调查加强了我们对深度检测、自动化调查和持续改进我们操作工具的承诺。可靠性不仅仅是关于在 bug 发生后修复——它是关于构建数据、工作流和技能，将不可能的问题转变为可诊断和可解决的问题。
